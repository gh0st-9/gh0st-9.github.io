Shiro550-反序列化

简介

Apache Shiro 是 Java 的一个安全框架。Shiro 可以非常容易的开发出足够好的应用，其不仅可以用在 JavaSE 环境，也可以用在 JavaEE 环境。Shiro 可以帮助我们完成：认证、授权、加密、会话管理、与 Web 集成、缓存等。

漏洞原理

Shiro 550 反序列化漏洞存在版本：shiro <1.2.4，产生原因是因为shiro接受了Cookie里面rememberMe的值，然后去进行Base64解密后，再使用aes密钥解密后的数据，进行反序列化。

所以构造该值为一条恶意java反序列化链，再对该值进行该密钥aes加密后进行base64加密，那么这时候就会去反序列化所传入的payload，执行攻击

利用链分析

漏洞点

在登录shiro时，如果点击了remember选项，就会生成一个Cookie来保存登录信息，实际上后端进行的操作是对用户登录信息进行序列化，然后进行AES加密，最后进行base64后生成Cookie。当你下次登录时，会携带所生成的Cookie进行登录

那么，如果能控制Cookie为恶意的序列化代码，然后通过相同的加密后传入，那么后端就会相应的解密反序列化，最后执行所传入的恶意代码

利用流程

全文搜索与Cookie或者RememberMe相关的类与方法

在CookieRememberMeManager中发现getRememberedSerializedIdentity(),大概执行的操作就是获取传入的Cookie，对其进行base64解密

往上跟，在AbstractRememberMeManager.getRememberedPrincipals中调用了getRememberedSerializedIdentity()

getRememberedSerializedIdentity()调用convertBytesToPrincipals()对获取到base64解密后的Cookie进行解析

跟进convertBytesToPrincipals()，操作很直观，对传入的参数进行解密后再调用deserialize()对其进行反序列化操作

跟进deserialize()，这里其实就是进行了一个反序列化操作，所以如果环境下有cc依赖或者其他的一些依赖，就可以直接利用来打漏洞了

主要来看看它解密的一个操作流程，跟进decrypt()

AbstractRememberMeManager.decrypt()

继续跟，CipherService.decrypt()，有两个参数，第一个参数为需要加密的字符串，第二个参数为加密所需要的Key

重点关注key的获取流程，如果能获取到key，就能通过相同的加密构造出恶意序列化代码

跟进JcaCipherService.decrypt(InputStream in, OutputStream out, byte[] key, boolean ivPrepended)，这里的解密密钥通过getDecryptionCipherKey()获得

发现返回的key是一个常量

往上找，看这个常量是在哪里赋值的

继续往上

往上找在哪儿调用了setCipherKey(),最终是在AbstractRememberMeManager()中找到key其实是DEFAULT_CIPHER_KEY_BYTES这个常量

而这个常量是一个固定的值，而它的加密算法，实际上就是一个AES算法

先拿cc6进行测试，结果发现恶意代码并没有执行成功，查看tomcat日志，发现如下报错

分析报错，“无法加载到Transformer”这个类

在cc6中我们使用了一个Transfomer数组去进行payload的构造

那么为什么加载不到这个类呢？跟进到它反序列化时的处理，在进行反序列化时，它并不是调用原生的反序列化操作，而是调用ClassResolvingObjectInputStream()进行一个处理，而ClassResolvingObjectInputStream()是shiro里面自定义的一个输入流。

跟进ClassResolvingObjectInputStream,问题出在resolveClass()

正常在反序列化中，是一定会经过resolveClass()处理的

而在ClassResolvingObjectInputStream类中重写了resolveClass()，在进行反序列化时就会调用重写的resolveClass()进行一个处理

对比一下原生的ObjectInputStream.resolveClass()

原生的resolveClass()是调用Class.forName进行一个类加载，而shiro重写的resolveClass()是调用shiro自定义的ClassUtils.forName及逆行类加载

跟进ClassUtils

可以发现ClassUtils加载类时是调用Classloader进行的一个类加载，而Classloader是不能加载数组类的

再往后对它的原理就不再具体深究了，因为后面的分析和安全其实就没有很大的关系了，是Tomcat内置的一些类加载的细节

exp的构造也就比较简单了，只要不出现数组类就行了

注意几个点，在构造的payload中我们利用了TransformingComparator类，可是TransformingComparator类不是CommonsColections4中的类吗？Shiro中是不带有cc4依赖的

其实这就涉及到了构造Exp时的一个技巧，只要保证序列化时的类是环境中存在的，之前怎样调用都是可以的

在Exp里后面部分通过反射修改了PriorityQueue类的值为BeanComparator类，所以在进行序列化时是不包含TransformingComparator类的

AES加密后执行，又出现了报错

找不到org.apache.commons.collections.comparators.ComparableComparator类

这个类不是cc3.2.1中的类吗，我们序列化的payload中是不包含cc3.2.1中的类的

实际上，问题出在BeanComparator,构造链中所调用的BeanComparator的有参构造，而其有参构造函数里调用了ComparableComparator

而ComparableComparator是cc3.2.1中的，解决方法也非常简单，调用BeanComparator类的另一个有参函数，该构造函数允许自己传入一个comparator，那传入一个cb或者jdk里面有的comparator就行了

找这个comparator也是十分好找的，因为要进行反序列化，所以要找一个继承serialize接口的，同时继承Comparator接口的。

有很多这样的comparator，这里就用AttrCompare.compare()就行了

无cc依赖链

在前面的链子中用到了cc2链中的PriorityQueue类去寻找compare，而此类依赖于org.apache.commons.collections4。
而在commons-beanutils包中存在一个类：org.apache.commons.beanutils.BeanComparator ，能够利用此类去寻找compare。
